# Job Tracker 开发心路历程与迭代思路

## 起点：从一个朋友的项目开始

开发这个项目的出发点，是参考了一个现成的求职追踪工具（`job-tracker` by Donzhu2020）。通过分析这个项目，我们提取了几个核心的工程思路：

1. **用 `python-jobspy` 免登录抓取 LinkedIn**：它调用的是 LinkedIn 未公开的 guest API，不需要账号和 API Key，直接用关键词和位置搜索职位。
2. **两层去重机制**：URL 精确去重 + 标题模糊去重，防止同一职位反复出现。
3. **定时调度**：Shell 脚本 + cron，轻量不依赖框架。

但原项目有几个核心局限：没有匹配过滤逻辑（全量推送）、没有通知功能（输出到 Obsidian 文件）、打分系统硬编码了特定领域偏好，无法跨领域使用。

---

## 第一次迭代：设计匹配系统

原项目的关键词打分系统是针对特定领域硬编码的，一旦用户跨领域就失效。我们考虑了两种方案：

- **方案A（关键词打分）**：从用户简历中提取技能词，与职位描述对比，命中一个词加5分，超过阈值则推送。简单直接，零 API 成本，适合方向固定的求职者。
- **方案B（AI 判断）**：把职位描述和用户的自然语言需求描述都扔给 AI，让它直接判断"推不推送"，不需要分数，更适合跨领域或转行的场景。

**迭代决策**：做成可选项。两种模式共存，用户根据自身情况在配置里切换。AI 模式下不设分数，改为让用户用一段话描述自己的求职要求，AI 直接返回推送/不推送的判断。

---

## 第二次迭代：AI Provider 解耦

最初 AI 模式只考虑接入 Claude API。但考虑到不同用户的偏好和已有账号不同，我们将 AI 调用层抽象出来，支持 Claude、OpenAI、DeepSeek 三种 provider，调用方式几乎相同（DeepSeek 兼容 OpenAI SDK），用户只需填入对应的 API Key 和 provider 名称即可切换，核心 prompt 逻辑完全复用。

---

## 第三次迭代：存储方案演进

最初考虑本地 SQLite 或 Google Sheets 二选一。但讨论后意识到这两者不应该是互斥的：

- **本地 SQLite 是主存储**：速度快、可靠、零依赖，去重逻辑以本地为准。
- **Google Sheets 是云端备份**：可选开启，保存一份镜像，方便多设备查看历史记录。

关于存储切换时历史数据的迁移问题：做数据迁移逻辑会增加大量复杂度，但实际使用中几乎不会切换，即便切换了重复推送几条也无伤大雅。最终决定不做迁移，切换时发一条 TG 提示即可，避免过度设计。

---

## 第四次迭代：通知系统设计

通知系统经历了较多的设计讨论，主要围绕以下几个维度：

### 通知方式
不强绑 Telegram，支持 Email 作为可选渠道，`channels` 配置里各自有 `enabled` 开关，互不干扰。

### 推送时机
两种模式满足不同用户习惯：
- **实时模式**：抓到匹配职位立即推送，适合对时效性要求高的场景。
- **定时模式**：职位暂存，到用户指定的时间点统一推送（可配置多个时间），适合不想被频繁打扰的用户。

### 免打扰
用户可配置免打扰时段，时区根据 IP 自动识别，解决跨时区找工作的场景（如在美国找欧洲职位）。免打扰期间的职位可选择"结束后统一推送"，保证不遗漏。

---

## 第五次迭代：健壮性增强

基础功能跑通后，补充了几个让系统更可靠的机制：

- **抓取失败重试**：最多重试 N 次，间隔可配置，连续失败后暂停并推送告警。
- **恢复通知**：恢复正常后主动推送"恢复正常"消息，让用户知道系统又活了。
- **每日心跳**：每 24 小时推送一次"运行正常"确认，防止系统静默挂掉而用户毫不知情。
- **间隔过短警告**：抓取间隔低于 30 分钟时主动告警，提示用户 LinkedIn 封禁风险。
- **Sheets 同步失败提示**：Sheets 同步失败不影响本地功能，但会通知用户检查配置。

---

## 第六次迭代：用户体验 - Web UI

所有配置项整理下来有几十个字段，直接让用户改 JSON 文件体验很差。考虑了三种方案：
- TG Bot 内配置：依赖 TG 连接状态，按钮交互对于这么多字段不够直观。
- 命令行交互：简单但不友好。
- **本地 Web UI（最终选择）**：Flask + 纯 HTML，无前端框架依赖，分 Tab 展示所有配置，表单填完点保存写入 config.json，同时提供"启动 Job Tracker"按钮。

为了让用户双击就能进入配置界面，在 macOS 桌面创建了 `.command` 快捷方式，首次运行自动安装依赖，后续直接启动。

---

## 第七次迭代：职位数据可视化管理

基础功能稳定后，提出了在 Web UI 里直接查看和管理职位数据的需求，而不是依赖 TG 记录或外部工具。这引出了几个设计决策：

**数据库 schema 扩展**：在原有的 jobs 表基础上新增字段——`starred`（收藏）、`status`（投递状态）、`notes`（备注）、`deleted`/`deleted_at`（软删除）。选择软删除而非直接删除，是为了支持垃圾箱恢复功能，同时也保留了去重记录的完整性。

**排序与筛选**：用户对排序维度的需求是多样的（时间、分数、公司名），因此做成可配置的下拉选择，而不是固定排序。筛选同理——公司名模糊搜索、状态精确筛选、日期范围筛选、仅看收藏，四个维度组合使用。

**导出设计**：导出支持 CSV 和 JSON 两种格式。CSV 面向用户直接用 Excel 打开；JSON 面向程序对接，比如后续的简历生成器。导出直接从浏览器下载，不依赖服务器文件系统。

**垃圾箱**：删除操作设计为软删除（标记 `deleted=1`），在垃圾箱 Tab 里可以恢复或永久删除，防止误操作。

---

## 第八次迭代：简历生成器接口预留

在讨论下一个程序（简历定制生成器）时，提前在 Job Tracker 里预留了对接接口，避免后续需要大改代码。

**接口设计的核心原则**：
- 匹配分数对简历生成器没有意义，不传
- 职位链接对用户投递很关键，必须传
- 完整 JD 描述是简历生成器最需要的，必须传
- 触发时机必须是**实时**的，不受定时推送影响

**输出方式两选一**：
- **本地文件**：适合简历生成器跑在同一台 Mac，监听目录有新文件就处理
- **Webhook**：适合简历生成器部署在其他机器或云端，接收 POST 请求

这个接口默认关闭，用户启用并配置输出方式后生效，对现有流程零影响。

---

## 整体架构演进总结

```
v1：抓取 → 全量输出到文件
v2：抓取 → 关键词匹配 → TG 推送
v3：抓取 → 双模式匹配（关键词/AI）→ 多渠道推送
v4：抓取 → 匹配 → 智能调度（实时/定时/免打扰）→ 多渠道推送
v5：抓取（含重试/告警）→ 匹配 → 调度 → 推送 + 心跳
v6：Web UI 配置 + 桌面快捷方式 + 完整文档
v7：职位列表 Tab（查看/排序/筛选/收藏/状态/备注/垃圾箱/导出）
v8：简历生成器输出接口（文件/Webhook，实时触发）
```

每一次迭代都是由一个具体的用户需求或痛点驱动的，而不是预先设计好的。这是这个项目在工程上最重要的一个特点。

## 第九次迭代：语义匹配升级 + 关键词别名学习系统

### 问题一：AI 匹配 prompt 语义太窄

早期 AI 模式的 prompt 过于依赖用户原话。用户说"软件工程师"，但职位写的是"Software Developer"——AI 可能因为字面不同而判断为不相关。

更大的问题是**多语言**：用户在香港找工作，JD 可能混合中英文，AI 需要跨语言理解语义等价。

**修复**：重写 AI 匹配 prompt，明确加入：
1. **语义优先**：Developer / Programmer / SWE 等同于软件工程师
2. **跨语言等价**：同一职位的中英文写法都算匹配
3. **上下位概念**：用户要"工程师"，职位写"高级工程师"也推送（反过来不一定）
4. **60% 门槛**：核心需求满足过半即推送，避免过严过滤
5. 原来硬编码的 `SYNONYM_GROUPS` 英文近义词列表一并删掉——这种"手工维护词表"的思路从根本上是错的，覆盖面永远不够

### 问题二：关键词搜索覆盖面不够

`python-jobspy` 通过关键词搜索 LinkedIn，关键词越精准覆盖越广。但用户只配置了"Software Engineer"，就只搜这三个字。
实际上 LinkedIn 上相同职位可能写成"Developer"、"SWE"、"开发工程师"——全部漏掉了。

**方案A（被否定）**：维护一个硬编码的近义词表。问题是领域依赖性太强，换个职位方向立刻失效，还不支持多语言。

**方案B（最终方案）**：AI 生成 + 本地缓存：
- 每个关键词调用一次 AI，生成该职位的多语言近义词列表
- 结果缓存到 `data/alias_cache.json`，格式为 `{关键词: {语言代码: [别名列表], last_updated: ...}}`
- 后续搜索直接用缓存，不再每次调 AI
- 过期时（超过 N 天）后台静默刷新

### 语言感知设计

学什么语言的别名，由用户的求职地区决定：
- 在英美加澳求职 → 只学英文别名（英文已是默认）
- 在香港/新加坡求职 → 同时学英文 + 中文别名
- 在德国求职 → 英文 + 德文别名

**英文永远是默认语言**，不管在哪里找工作，英文别名必学。

**按需学习**：如果抓取到了预期之外语言写的 JD（比如在美国找到了意大利语职位），就针对那个语言单独后台学习一次，不影响主流程。

### 触发机制

| 触发时机 | 说明 |
|---------|------|
| 用户新增关键词时 | 立即后台学习（该关键词） |
| 每次抓取前 | 检查是否有过期缓存，过期则后台刷新 |
| 抓取结果中出现新语言 | 后台补学该语言的别名 |

所有学习都在**后台线程**完成，不阻塞主流程，不影响推送时效。

### Token 消耗控制

对一个用户关心的核心点单独说明：每个关键词每种语言最多存 5 条别名（AI 会被 prompt 约束），而且一次学习后缓存 N 天（默认7天）才重新调 AI。日常运行几乎不消耗额外 Token。

## 第十次迭代：跨平台 + 数据路径独立化

### Windows 支持

`.command` 是 macOS 专属格式，Windows 用户无法启动。添加了 `Double Click to Start_Windows.bat`，逻辑与 Mac 版对等：虚拟环境激活、依赖安装、服务启动、浏览器打开，使用 Windows 原生命令（`netstat` 替换 `lsof`，`.venv\Scripts\activate.bat` 替换 Shell source）。

同步将 macOS 启动脚本重命名为 `Double Click to Start_Mac.command`，两者命名风格一致。

### 数据路径独立化

存储路径历史上用了相对当前工作目录的路径（`~/.job-tracker/`），导致从不同目录启动程序时，数据库路径可能不一致。

修复：所有路径解析统一基于 `Path(__file__).parent`（程序文件所在目录），相对路径在配置中原样保留，运行时动态拼接到程序目录。具体改动：
- `storage.py`：`get_db_path()` 用 `BASE_DIR / raw_path` 解析
- `main.py`：日志文件写到 `data/job-tracker.log`
- `notifier.py`：文件输出目录相对程序目录解析

移动整个文件夹、分享给朋友、换用户名——数据全部跟着走，不再需要重新配置路径。

## 最新修复（v9）：多语言支持 + 简历解析升级

### 问题：关键词模式对非英文 JD 完全失效

用户遇到了一个实际场景：收到意大利语和法语的职位描述，关键词模式匹配分数为零，导致明明相关的职位被过滤掉了。

根本原因：`SKILL_MASTER_LIST` 全是英文，而 JD 里写的是 `analisi dei dati`、`gestion des risques` 这样的词，自然匹配不上。

**修复方案：**
1. `SKILL_MASTER_LIST` 新增意大利语、法语、中文对应词条
2. 加入轻量级语言检测（`detect_language`），当检测到非英文 JD 且匹配分数低时，在推送理由里提示用户切换 AI 模式
3. AI 模式的 prompt 明确说明"直接理解原文，无需翻译"，避免 AI 自行翻译导致的语义损失

核心教训：**关键词模式天生不适合多语言场景**。对于跨语言求职用户，AI 模式才是正确答案，我们能做的是在关键词模式下给出明确提示，而不是硬撑。

### 简历解析：PyPDF2 → MarkItDown

PyPDF2 对分栏 PDF、表格简历的解析质量较差，读出来的文字顺序可能错乱。

换成微软的 `MarkItDown` 后，解析质量明显提升，同时保持了 PyPDF2 和 python-docx 作为降级备选，安装了 MarkItDown 就优先用，没装也不影响运行。

## 启动脚本优化（v10）：依赖管理更健壮

### 问题：venv 存在但依赖丢失时无法启动

遇到一个典型场景：`.venv` 目录存在，但内部依赖包未安装（例如清理数据后重新分发、或从他人处拷贝程序文件夹）。原启动脚本只检查 `.venv` 目录是否存在，存在就直接启动，导致 `ModuleNotFoundError: No module named 'flask'`，浏览器显示"can not connect to the server"。

另外发现 `requirements.txt` 中的包名有误：`python-jobspy` 在 PyPI 上已更名为 `jobspy`，首次安装必然失败。

**修复方案：**

1. **`requirements.txt`**：`python-jobspy` → `jobspy`
2. **启动脚本（Mac & Windows）**：将依赖安装从"只在首次创建 venv 时执行"改为"每次启动都执行"

```bash
# 新逻辑
创建虚拟环境（如不存在）→ 激活 → pip install -q -r requirements.txt（每次）
```

pip 对已安装且版本匹配的包会瞬间跳过，不重新下载，对用户体验无影响。

**覆盖的场景：**
- 首次启动（正常安装）
- venv 存在但依赖丢失（自动修复）
- 移动或分享文件夹后启动（自动修复）
- requirements.txt 更新后（自动补装新依赖）

## UI Bug 修复（v11）：删除按钮失效

### 问题：点 × 删关键词，反而被当成新增

用户点击关键词标签上的 × 删除按钮，结果关键词没有删掉，反而行为异常。

根本原因有三层，叠加导致：

1. **`<button>` 默认 `type="submit"`**：浏览器把删除按钮当成提交按钮，触发了意外的表单行为
2. **点击事件冒泡**：× 按钮的点击冒泡到外层容器，容器的 `onclick` 把焦点转移到输入框，造成干扰
3. **`keydown` 监听器不断叠加**：原来每次删除都重新调用 `initTagInput`，而 `initTagInput` 每次都往 input 上 `addEventListener`，不移除旧的。删了 N 次后，input 上挂了 N+1 个监听器，按一次 Enter 触发 N+1 次添加

**修复方案：**

- `initTagInput` 改为维护模块级 `_tagData` 对象，不再每次重建闭包
- 删除按钮改用 `document.createElement('button')` 创建，设 `type="button"` + `e.stopPropagation()`
- 用 `input.dataset.listenerAttached` 标记，`keydown` 只绑一次

同步检查并修复了时间标签（`renderTimes`）和别名标签（`_drawAliases`）中相同的 `type` 缺失问题。

**教训**：`<button>` 在 HTML 中默认是 `type="submit"`，在任何非表单场景下都应显式写 `type="button"`，是一个极易忽略的坑。

## 运行状态与控制（v12）：WebUI 实时显示 + 停止按钮

### 背景

原来 WebUI 只有一个「启动」按钮，启动后完全不知道 Tracker 是否还在跑，也没有办法从界面停止它。

**新增功能：**

1. **实时状态指示器**：Header 右侧显示绿色呼吸灯（运行中）/ 灰色（已停止），每 5 秒自动轮询 `/api/status`
2. **停止按钮**：运行时显示红色停止按钮，停止后切回启动按钮
3. **配置变更检测**：保存时若关键词或匹配模式有改动，且 Tracker 正在运行，弹窗询问是否立即重启生效（否则下次抓取周期自动生效，因为 `main.py` 每轮都会 `load_config()`）

**实现方式：**
- 后端用 `data/tracker.pid` 文件记录进程 PID，`/api/status` 用 `os.kill(pid, 0)` 探活，`/api/stop` 发 SIGTERM

## 别名学习联动关键词（v13）：新增关键词自动触发学习

### 背景

在搜索设置里添加或删除关键词后，别名学习 Tab 的缓存视图不会同步，用户需要手动切换过去再刷新。

**修复：**

`_syncAliasesWithKeywords` 增加两个动作：

1. **自动触发学习**：检测到新增关键词时，若别名学习已启用且填写了 API Key，立即调用 `/api/alias_learn` 在后台学习，3 秒后刷新缓存视图
2. **即时刷新缓存**：若别名学习 Tab 当前处于打开状态，切换关键词后缓存视图立即刷新

未启用别名学习或未填 API Key 时不触发，行为与之前一致。

## Python 兼容性全面加固（v14）：支持 Python 3.9+ 并强化启动环境检测

### 背景

用户重启机器后双击启动，出现：

```
TypeError: unsupported operand type(s) for |: 'type' and 'NoneType'
```

根本原因：`setup.py` 中使用了 `int | None` 联合类型注解语法，这是 Python 3.10 引入的特性，在系统默认的 Python 3.9 上会在运行时报 `TypeError`。

### 修复方案

**分两层解决：**

**1. 代码层（根本修复）：**

所有 Python 源文件（`main.py`、`setup.py`、`storage.py`、`scraper.py`、`matcher.py`、`notifier.py`、`scheduler.py`、`alias_learner.py`）统一在文件顶部加入：

```python
from __future__ import annotations
```

这个 import 从 Python 3.7 起就支持，作用是将所有注解改为懒求值（字符串形式），使得 `int | None`、`list[dict]`、`X | Y` 等任何注解写法都不会在运行时求值，从根本上消除所有注解相关的版本兼容问题。以后在这些文件里可以放心使用任何现代注解语法。

`setup.py` 额外将 `int | None` 改为 `Optional[int]`（两者等价，后者更保守），作为代码可读性的额外保障。

**2. 启动脚本层（用户体验）：**

`Double Click to Start_Mac.command` 新增 Python 环境检测逻辑：
- 按优先级检测 `python3.13 → 3.12 → 3.11 → 3.10 → 3.9 → python3`，找到第一个 ≥ 3.9 的版本即用
- 若无合适版本，检测到 Homebrew 则自动执行 `brew install python3`
- 若 Homebrew 也没有，打印两种手动安装方式并保持终端开着
- venv 创建失败、pip 安装失败均单独捕获并给出可操作的提示

## 启动可靠性修复（v15）：Shell 层等待服务就绪后再开浏览器

### 问题

重启电脑后双击启动脚本，浏览器不弹出，手动访问地址也无法连接。

根本原因在于两个设计缺陷叠加：

1. **浏览器由 Python Timer 负责打开**：`setup.py` 里用 `Timer(1.0, open_browser)` 在启动后 1 秒打开浏览器。Flask 启动通常不止 1 秒（pip 安装 + import + 绑定端口），计时器比服务就绪早，`webbrowser.open()` 指向的地址尚不可用
2. **启动失败无任何反馈**：若 Flask 因任何原因（import 报错、端口冲突）崩溃，终端窗口直接关闭，用户看不到任何错误信息

### 修复方案

**`Double Click to Start_Mac.command`**：
- 改为在 Shell 脚本中后台启动 `python3 setup.py`
- 主进程每秒用 `curl` 轮询 `http://localhost:8080`，确认服务真正可用后用 macOS 原生 `open` 命令打开浏览器
- 最多等待 30 秒；期间每次循环同时检查子进程是否存活，若进程已崩溃立即报错并保持终端开启（让用户看到报错信息）
- 如果服务已在运行（重复双击），直接开浏览器并退出，不重复启动

**`setup.py`**：
- 删除 `Timer(1.0, open_browser)` 及相关 import（`webbrowser`、`Timer`），浏览器打开职责完全交给 Shell 层

**效果**：
- 双击 → 终端显示 `等待服务就绪.......✅` → 浏览器准时弹出
- 启动失败时终端保留错误日志，用户可以看到具体原因
- 重启机器后行为与日常使用完全一致

## 多地区求职支持 + AI 匹配逻辑重构（v16）

### 问题一：单地区设计无法应对跨城市求职

原来 `config.json` 只支持一个 `location` 字段和一个全局 `user_prompt`，所有关键词混在一起用大 OR 查询一个地区。用户同时在德国找电气工程、在米兰找金融咨询时，只能把两个城市的需求混在同一条 prompt 里，结果 AI 要用同一套标准评判完全不同的职位类型，逻辑上就是错的。

**更深的问题**：关键词全量 OR 查询时，`"EE"`（电气工程师缩写）会匹配到无数公司名缩写，`"Management consultant"` 会拉回所有带"管理"字样的职位。LinkedIn 返回的结果里已经包含了大量无关职位，AI 再怎么筛也有漏网之鱼。

### 解决方案：per-location 独立配置

`config.json` 从单 `location` 字段升级为 `locations` 数组，每个地区独立配置：

```json
"locations": [
  {
    "name": "德国",
    "location": "Frankfurt, Germany",
    "keywords": ["电气工程师", "Electrical Engineer"],
    "keyword_aliases": { ... },
    "user_prompt": "德国的电气工程类相关工作",
    "enabled": true
  },
  {
    "name": "米兰",
    "location": "Milano",
    "keywords": ["金融咨询", "IT咨询", "strategy consulting"],
    "user_prompt": "米兰的金融咨询、IT咨询或战略咨询类工作",
    "enabled": true
  }
]
```

`scraper.py` 重构为遍历所有启用的地区分别搜索，每条抓回的 job 携带 `_user_prompt` 和 `_location_name` 字段。`matcher.py` 的 `match_with_ai()` 优先读取 `job["_user_prompt"]`，每条职位用它所属地区的偏好来判断。向下兼容旧版单 `location` 格式，老配置不需要改。

通过 `"enabled": false` 可以随时暂停某个地区而不丢失配置。

### 问题二：AI 匹配误推"调酒师"和"航天工程师"

运行日志出现了完全不相关的职位被推送。根本原因是 AI 判断 prompt 有两个设计缺陷：

1. **"60% 需求满足就推送"**：AI 会用"公司从事相关行业"来凑满足率，比如一家工程公司招了个调酒师，AI 认为"地点对，公司行业对，满足 60%"
2. **没有明确区分硬性要求和软性要求**：把"学历不够""签证不对"和"工作年限少两年"当成同等权重来评判

### 解决方案：两层判断结构

重写 AI system prompt，明确分为硬门槛和软条件两层：

**第一层（硬门槛，任一命中直接拒绝）：**
- 地区不符
- 职位大类跨行业差距过大（用户要电气工程师，职位是调酒师、护士、司机）
- 学历有明确 required/mandatory 且明显达不到
- 签证/国籍限制（"no sponsorship"、"must be EU citizen"）
- 语言硬性要求（"mother tongue required"）
- 明写的年龄/性别要求

**第二层（软条件，宽松放行）：**
- 工作年限不足：要5年有3年？推送，HR 经常弹性处理
- 技能部分缺失：可以学
- preferred / nice to have 类表述：根本不算要求

**总原则写进 prompt**：拿不准是否应该拒绝时，推送。让用户自己判断是否值得申请，不要替用户放弃机会。

这个改动解决了两类误判：过松（调酒师/航天工程被推）和过严（工作年限差一两年就被过滤）。

## UI 全面升级（v17）：多地点卡片 + 全局交互打磨

### 背景

v16 完成了后端和配置的多地区支持，但 Web UI 还停留在旧设计：一个地点字段 + 一个全局关键词输入框。用户在 UI 里完全看不到"德国 → 电气工程"和"米兰 → 金融咨询"是两套独立配置，也没有办法从界面管理它们。配置文件已经升级了，UI 却没跟上——这是最典型的"内外不一致"问题。

### 搜索面板重新设计：多地点卡片

原来的"位置设置"卡片（一个地点输入框 + 半径）和"搜索关键词"卡片（全局关键词 tag input + alias 编辑器）完全删除，替换为一个统一的"搜索地点"区块。每个地点是一张独立的卡片，包含：

- **地点名称**：自由命名（如"德国"、"米兰"）
- **关键词 tag input**：每个地点独立一套关键词，互不干扰
- **城市 / 半径**：可折叠，细节不占主要视觉空间
- **AI 提示词**：每个地点可以写不同的自然语言求职描述，直接传给 AI 匹配
- **启用开关**：iOS 风格 toggle，关闭后该地点跳过爬取，配置保留
- **alias 编辑器**：可折叠，管理关键词的同义词/变体

卡片左侧用 `LOC_COLORS` 数组（蓝/绿/橙/紫/红/青）做彩色 index badge，一眼区分多个地点。底部"添加搜索地点"按钮动态追加新卡片。

### JS 状态管理重构

旧系统用全局 `_aliases` 对象存所有 alias，新系统改为 `_locAliases[i]` 数组，每个地点索引对应一个独立 alias 列表。tag input 的初始化函数 `initTagInput` 增加了 `onChangeFn` 回调参数，关键词变化时自动触发 `_syncLocAliasesWithKeywords(i)` 同步清理已删除关键词对应的 alias。

每张卡片的 DOM ID 采用 `loc-${i}-*` 命名规范，避免同页多实例冲突。

### `populate()` 和 `collectConfig()` 向下兼容

新的 `populate()` 先检查 `s.locations` 数组是否存在，不存在就用旧格式的 `s.location` / `s.keywords` 字段构造一个单地点数组，老配置不需要改动就能平滑迁移。`collectConfig()` 统一写 `locations` 数组格式，同时删除旧的顶层 `location` / `radius_km` / `keywords` / `keyword_aliases` 字段，保持配置文件干净。

### 其他 UI 细节

- **嵌套 label bug**：`_renderLocCard` 模板里"启用"toggle 的外层容器原来是 `<label>`，内部又有 `<label class="toggle">`——HTML 规范不允许 label 嵌套 label，浏览器行为不可预期。外层改为 `<span>` 修复。
- **"抓取频率"与"全局时间设置"合并**：原来两张独立卡片，内容少且相关，合并为"抓取频率与时间"一张，减少页面纵向长度。
- **header subtitle**："自动爬取 · AI 匹配 · 实时推送"，几个字概括整个系统的工作流，第一次打开的用户也能快速理解。
- **关键词数量 badge**：每张地点卡片 header 右侧显示当前关键词数量，折叠卡片后也能一眼看出配置状态。

### 感想

这次改动最大的体会是：配置文件的数据结构和 UI 的状态模型必须同步演进。v16 重构了 `config.json` 的 `locations[]` 结构，但 UI 没跟上，导致用户必须手动编辑 JSON 才能用多地点功能——等于功能做了但用不上。UI 重构本质上是把配置文件的层级结构"投影"到界面上，每一层数据对应一个视觉单元（卡片）、一套交互（tag input、toggle、折叠）。

多实例 tag input 的 ID 命名规范（`loc-${i}-keywords-input`）、回调函数设计（`onChangeFn`）这些看起来是细节，但如果一开始没想清楚，后期 debug 状态污染会很痛苦。

## Bug 修复与 UI 细节打磨（v18）

### Bug：时区手动输入框永远不显示

「通知设置 → 免打扰 → 时区」有两个 radio：自动（根据 IP）/ 手动填写。切换到"手动填写"应该展开输入框，但实际上什么都不发生。

根本原因是三处遗漏并存：

1. **`updateVisibility()` 没有处理 `quiet_tz` radio**：函数里有 `quiet-options`、`sheets-options` 等十几个 show/hide 判断，唯独漏了 `quiet-tz-field`。
2. **事件监听器列表里没有 `quiet_tz`**：底部的 `addEventListener` 串只绑定了其他 radio，没有绑定 `input[name="quiet_tz"]`，所以改变 radio 根本不触发 `updateVisibility`。
3. **`collectConfig()` 里 timezone 硬编码为 `'auto'`**：即便用户填了手动时区，保存时也被丢弃，直接写死 `timezone: 'auto'`。

同时 `populate()` 也没有把 `config.notification.quiet_hours.timezone` 还原到 radio 上，每次刷新页面 radio 总是重置为"自动"。

**修复**：
- `updateVisibility()` 加了 `quiet_tz` radio 的 show/hide 判断
- 事件监听器列表补了 `querySelectorAll('input[name="quiet_tz"]').forEach(r => r.addEventListener('change', updateVisibility))`
- `collectConfig()` 改为读 radio 值 + input 框内容，正确组装 `timezone`
- `populate()` 加了对应的 radio 还原逻辑

这个 bug 的特点是"UI 是完整的，逻辑是缺失的"——输入框有了、radio 有了、只是没有任何代码把它们串联起来。和界面没显示、报错不一样，这类 bug 用户往往操作一遍发现没效果，但不知道哪里错了。

### 感想

v18 的改动很小，但体感上很有价值。时区是一个"大部分用户不会碰，少数用户非常依赖"的设置，对于跨时区工作的用户来说，手动设置时区才能让免打扰功能真正有用。功能做了一半放在那里，比没有这个功能更糟糕——用户会以为自己设置了却没有生效，反而失去信任感。

做完这次修复之后，顺手把 `collect → save` 路径上所有依赖状态的字段都又扫了一遍，确认没有同类型的"UI 有但逻辑断开"的情况。

## 职位卡片交互完善 + HR 联系人多引擎搜索（v19）

### 背景

Job Tracker 推送的职位通知一直是纯文本，没有按钮。telegram_poller.py 和 hr_finder.py 代码早就写好了，但一直没有触发——因为按钮出现的条件是 `job.get("description")` 非空，而 jobspy 默认不抓职位正文，所有匹配职位的 description 字段全是空的。

### 问题一：按钮从不出现

最初的设计思路是在抓取时就把职位 JD 一并存下来（开启 `linkedin_fetch_description=True`），这样推送时 description 有值，按钮才会出现。

但这个方案有两个问题：
1. **效率浪费**：50 条结果每条都要单独请求 LinkedIn 详情页，大量用户不感兴趣的职位白白消耗请求配额
2. **数据不新鲜**：用户可能几小时后才看到推送，期间 JD 可能已经更新过

正确的做法是：**按钮出现只需要有职位链接，JD 在用户点按钮时实时抓取**。

改动：
- `notifier.py`：按钮条件从 `job.get("description")` 改为 `job.get("link")`
- `telegram_poller.py`：新增 `_fetch_jd(link)`，点按钮时调用 Resume Generator 的 `/api/fetch_jd` 接口实时获取 JD；LinkedIn URL 单独走 `jobs-guest API`（`/jobs-guest/jobs/api/jobPosting/{job_id}`）绕过登录墙
- `scraper.py`：移除 `linkedin_fetch_description=True`

### 问题二：HR 联系人搜不到

hr_finder.py 原来只用 DuckDuckGo Lite，但 DDG 对 `site:linkedin.com/in/` 这类查询越来越严格，频繁返回首页或触发限流。

**改成六引擎并行**：Bing、DuckDuckGo、Brave、Yahoo、Yandex、SearXNG（开源元搜索引擎，聚合 Google/Bing/DDG 结果）。并发请求，结果按 LinkedIn slug 去重合并。

同时加了**公司级缓存**（6 小时 TTL），搜索结果存到 `data/hr_cache/`。同一公司在缓存期内重复点按钮不会重复请求搜索引擎，避免触发限流。

原来硬编码最多显示 5 个联系人，改为爬到几个显示几个，完整信息比数量限制更重要。

### 关于名字显示

搜索引擎返回的 HTML 里，LinkedIn URL 往往藏在 JavaScript 渲染结果或图片标签里，snippet 文本没有 "名字 - 职位 | LinkedIn" 这样的结构。直接从 LinkedIn profile 页取名也行不通——未登录请求会被 redirect 到登录页。

兜底方案：从 LinkedIn slug 推断可读名字（`florine-v-27107a95` → `Florine V`），去掉末尾像哈希的字符串，各段首字母大写。不精确，但比显示 `--` 好，用户点进去能看到真实信息。

### 感想

这次改动的核心是一个设计原则：**不要提前做用户不一定需要的工作**。提前抓 JD 看起来合理，但用户没点按钮就白抓了，而且拿到的不一定是最新的。等用户真正感兴趣了再实时去拿，逻辑更干净，资源消耗更低。

多引擎搜索是另一个典型的"多做一点保险"的思路。单一引擎在没有 API Key 的情况下极不可靠，加几个引擎的边际成本几乎为零（并发），但容错能力大幅提升。缓存解决了重复触发的问题，让整个系统在不稳定的第三方服务上保持相对可靠。

## TG 状态同步 + 地区黑名单 + JD 学习 + UI 优化（v20）

### 背景

随着多地区功能落地和用户真正开始使用，暴露了几个体验断层：TG 端和 Web 端状态不同步、全局黑名单无法精细控制到单个地区、没有办法直接"喂" JD 给程序学习、Web 界面在功能增多后显得拥挤。v20 一次性补齐这四块。

### TG 不感兴趣 → Web 状态同步

**问题**：用户在 TG 点了"👎 减少推荐 → 选择原因"，`save_feedback()` 确实记录了，但 Web 端的职位列表里这条职位状态还是"新"，视觉上看不出来已经被标记过。

**原因**：`_handle_dislike_reason_callback` 只调用了 `save_feedback`，没有同步更新 `jobs` 表的 `status` 字段。

**修复**：在 `telegram_poller.py` 的 `_handle_dislike_reason_callback` 里追加 `update_job(jid, {"status": "disliked"})`。

同时 Web 端配套：
- 新增 `.status-disliked` 红色标签
- 状态筛选下拉新增"不感兴趣"选项
- 已标记不感兴趣的职位卡片显示为半透明（`opacity: 0.45`），鼠标悬停时稍微恢复（`0.75`），视觉上区分已处理/未处理
- 不感兴趣的职位隐藏"👎"按钮，避免重复点击

### 地区级黑名单

**问题**：原来的黑名单是全局的（`config.search.blacklist`），一旦两个地区对某个公司/词的处理策略不同（例如某公司在德国有合适职位，在米兰的岗位方向不对），就没有办法精细控制。

**方案**：在每个地区配置里新增 `blacklist.companies` + `blacklist.keywords`，与全局黑名单**合并生效**（不是覆盖）。

**数据流**：
1. `config.json` 每个 location 新增 `blacklist` 字段
2. `scraper.get_locations()` 透传 `blacklist` 到 loc_config
3. `scraper.apply_blacklist(jobs, global_config, loc_blacklist=None)` 新增第三个参数，把地区黑名单 append 到全局黑名单后合并过滤
4. `_scrape_one_location` 传入 `loc_config.get("blacklist")` 调用

**Web UI 支持**：
- 偏好设置页每个地区卡片新增「🚫 该地区黑名单」区块，两列分别输入屏蔽公司和屏蔽关键词（每行一条）
- 全局卡片也新增「🚫 全局黑名单」区块
- `savePreferences` / `api_preferences_post` 同步读写 `blacklist` 字段
- `_parseLines(id)` 工具函数：把 textarea 内容按行拆分、去空行

### JD URL 学习

**功能**：在偏好设置的每个地区卡片里，新增一个 URL 输入框 + "从JD学习"按钮。用户粘贴一个招聘帖的网址，程序抓取页面内容后调用 AI 提取职位要求，以「替换」或「追加」的方式更新该地区的求职描述。

**技术**：
- `_fetch_jd_text(url)` 用 `requests` 抓取页面，用 Python stdlib 的 `html.parser` 剥离 HTML 标签（不引入 BeautifulSoup 新依赖），截取前 4000 字符
- `/api/preferences/learn-from-jd` 接口：接受 URL + 当前描述 + 地区名，调用 `matcher._call_ai` 提取要求
- 前端展示建议结果，用户选择"替换"/"追加"/"忽略"

注意：LinkedIn 等需要登录的页面会抓取失败，这种情况用户可以直接粘贴 JD 文字。

---

### UI 整体优化

偏好设置页的卡片结构做了调整：

- **新卡片结构**：每个地区卡片分为 `pref-card-header`（灰底标题栏）+ `pref-card-body`（内容区），内容区内部再分 section，用 `pref-section-label`（小型大写字母标题）作为视觉分隔
- **黑名单区块**：红色浅底（`#fff8f8`）+ 红色边框，与求职要求描述区块形成视觉区分，一眼识别"这里是过滤设置"
- **职位卡片**：已标记不感兴趣的职位半透明展示，不显示"👎"按钮，减少视觉噪音

---

### 感想

这次改动里最有意思的是黑名单的"合并"设计。最初直觉是"地区黑名单覆盖全局"，但仔细想想覆盖会让全局黑名单的效力消失——用户一旦给某地区设了黑名单，全局的就不管用了，这显然不对。合并才是自然的语义：全局规则始终生效，地区规则在此基础上叠加。两个列表拼接后统一过滤，实现简单，行为直觉。

JD 学习功能的设计取舍是不引入 BeautifulSoup。大部分求职网页的内容都在 `<p>` `<li>` `<span>` 里，用 `html.parser` 剥标签拿文本已经足够，多引入一个依赖的边际价值接近零，出错面反而变大。

TG 和 Web 状态同步这个问题，根本上是"写入点分散"导致的。反馈从 TG 写入 `user_feedback` 表，状态从 Web 写入 `jobs` 表，两张表没有级联。修复方式很直接：在 TG 侧写反馈时同步写 status。这个改动一行，但用户体验改善很明显——两个界面终于说同一种语言了。

## 联系人归属修复（v21）：在职验证 + 职位无关 + 子串 bug 修复

### 背景

用户反映推送的联系人有时来自其他公司（不是当前这条 job 的公司），而且只想要"现在在这家公司工作的人"，不在乎对方是什么职位。v21 把联系人搜索逻辑从根上重建。

---

### 问题一：旧查询搜的是"提到过"而非"在职"

旧查询：`site:linkedin.com/in/ "Deutsche Bank"`

只要页面上出现 "Deutsche Bank" 就命中——包括过去工作经历、关注了这家公司、评论了他们的招聘帖。

**修复**：查询改为 `site:linkedin.com/in/ "at Deutsche Bank"`。

LinkedIn 把每个 profile 的页面标题格式化为：
```
Name - Current Title at Current Company | LinkedIn
```
`at Company` 是 LinkedIn 当前在职职位的固定格式。搜 `"at Company"` 就是专门命中这种标题格式，比直接搜公司名精准得多。

---

### 问题二：snippet 导致跨公司污染

旧的 `_is_company_match` 同时检查 `title` 和 `snippet`。搜索引擎的 snippet 是自动生成的摘要，经常包含历史工作经历，例如：

```
"...曾任职于 Company A，现在 Company B 担任..."
```

这导致 Company A 的搜索结果里混入了当前在 Company B 工作的人。

**修复**：`_is_current_employee` 只检查 `title`，完全不看 `snippet`。

| 字段 | 内容 | 是否可信 |
|------|------|--------|
| title | `<title>` 标签，LinkedIn 固定格式 `Name - Title at Company` | ✅ 当前职位 |
| snippet | 搜索引擎自动摘要，可能包含任何文字 | ❌ 不可信 |

---

### 问题三：子串误匹配

旧的检查用 `f"at {company}" in title`，"at Google" 是 "at Google DeepMind" 的子串，导致 Google DeepMind 的员工被误认为 Google 员工。

**修复**：改用正则加负向前瞻：
```python
re.escape(f"at {company_lower}") + r"(?!\s+[a-z])"
```
`(?!\s+[a-z])` — 若 `at company` 之后跟着空白 + 小写字母，说明公司名没结束，整体失败。

| 情形 | 标题 | 公司 | 结果 |
|------|------|------|------|
| 精确匹配 | `at Google \| LinkedIn` | Google | ✅ |
| 子公司污染 | `at Google DeepMind \| LinkedIn` | Google | ❌ 正确拒绝 |
| 带地点 | `at Deutsche Bank, Frankfurt \| LinkedIn` | Deutsche Bank | ✅ |
| 带后缀 | `at Deutsche Bank AG \| LinkedIn` | Deutsche Bank AG | ✅ |

---

### 问题四：职位不重要，只要在职

旧逻辑优先推 HR/招聘岗位（`_is_hr_relevant`）。用户反馈职位根本不重要，只要确认当前在职就行。

**修复**：删除 `_HR_SIGNALS` 和 `_is_hr_relevant`，过滤逻辑简化为一条：

```python
contacts = [c for c in all_results if _is_current_employee(c, company)]
```

---

### 同步去掉 Brave / Yandex 引擎

这两个引擎只能从页面源码用正则抓出 LinkedIn URL，不提取页面标题。没有标题就无法验证在职，这类结果全部被 `_is_current_employee` 拒绝。继续发请求只是浪费，从引擎列表里移除。保留 Bing、DuckDuckGo、Yahoo、SearXNG（这四个都会提取页面标题）。

---

### 感想

这次问题的核心是"搜索结果的可信字段"判断失误。snippet 看起来信息丰富，但恰恰因为它包含的是自由文本，信噪比很低。title 反而因为是 LinkedIn 的固定格式而极其可靠。以后处理任何来自第三方的结构化数据，都应该先问：哪个字段是由数据源控制格式的，哪个字段是自由文本？前者优先信任，后者小心使用。

子串 bug 是一个经典的"没有用词边界"错误。`"at google" in text` 在字符串匹配层面永远无法区分 "at google" 和 "at google deepmind"——需要明确告诉程序"google 之后不应该有更多词"。正则的负向前瞻就是为此而生的。

---

## 地区配置拆分 + Bug 修复（v22）：城市 / 国家双输入框

### 背景

之前地区配置只有一个「城市/地区」输入框，用户直接填 `"Milano"`，结果 jobspy 把它识别成美国德克萨斯州的 Milano，而不是意大利米兰——导致抓到的职位全是美国的。根本原因是缺少国家信息，城市名存在歧义。

---

### 改动一：地区输入拆分为城市 + 国家

把单输入框改为两个并排的独立输入框：

- **城市**（如 Frankfurt、Milano）
- **国家**（如 Germany、Italy）

两者都填后，系统自动拼成 `"Frankfurt, Germany"` 传给 jobspy，消除歧义。

同时补齐了校验逻辑：城市/国家均不能为空、不能少于 2 字符、不能含数字、不能含逗号（逗号会破坏拼接格式）。国家为空时错误提示直接说明原因：「避免城市名歧义，如 Milano TX 与 Milano Italy」。

旧配置（只有 `location` 字段）自动兼容：按逗号拆分回填到城市和国家，迁移无感知。

---

### 改动二：Bug 修复批次

这次顺带修了几个积累的 bug：

**1. `config.json` 缺 city/country 字段**
原来 config 里只有 `location` 字符串，新 UI 读 `city`/`country` 拿到的全是空。修复：启动时自动从 `location` 拆分回填，同时发现米兰条目连国家都没有（`"Milano"` 无逗号），手动补上 `Italy`，location 修正为 `"Milano, Italy"`。

**2. AI 别名学习传入空 location**
`_syncLocAliasesWithKeywords` 里读的是 `loc-${i}-location` 这个 DOM ID，但 UI 改版后这个 ID 已经不存在了，永远拿到空字符串，AI 学别名时不知道是哪个城市。修复：改为读 `loc-${i}-city` + `loc-${i}-country` 拼合后传入。

**3. dislike 状态不持久化**
用户点「👎 不感兴趣」后，前端只是把卡片透明度调低，刷新页面就恢复原样。数据库里 `jobs.status` 字段从没被更新。修复：在 `api_feedback_post` 里补上 `update_job(job_id, {"status": "disliked"})`，反馈写入的同时同步更新职位状态，两个界面终于说同一种语言。

---

### 新增地区：Berlin + Vienna

在 config 里新增两个搜索地区：
- **Berlin, Germany**：关键词、AI 描述与法兰克福保持一致（扩大德国覆盖范围）
- **Vienna, Austria**：关键词、AI 描述与米兰保持一致（最初误写为意大利，已更正）

---

### 感想

城市名歧义这个问题很典型：用户填 `"Milano"` 完全合理，谁会想到程序会把它解析成德克萨斯？错误发生在"输入格式定义"阶段，却在"搜索结果"阶段才暴露，中间隔了很长的调用链。解决方式不是让用户记住"要写完整国家名"，而是把输入框拆开，让格式约束在 UI 层就生效。

Bug 2 里的失效 ID 是改版时漏掉的联动点。单一文件里的 UI 和逻辑耦合太紧，改了 HTML 结构没有同步更新 JS 里对应的 ID 引用，这类错误在大文件里很难一眼发现。拆文件是长期解法，短期只能靠 code review 时专门检查 ID 引用。

dislike 状态不持久化那个 bug 说明两个写入路径（TG 写 `user_feedback`，Web 写 `jobs`）没有形成闭环。每次给用户开放一个新的写入入口，都需要问：「这个操作应该影响哪张表？影响了吗？」而不是假设前端的视觉反馈等于数据已更新。

## 系统级 Bug 修复（v23）：八个隐藏缺陷

### 背景

这轮没有新功能，纯粹是一次系统性 code review 找出并修复了八个积累的隐藏 bug。大部分都是"跑着跑着才会暴露"的问题：竞态、数据泄漏、字符串拼写不一致。

---

### Bug 1：`"keyword"` vs `"keywords"`——一个字母的差价

`notifier.py` 里判断显示格式时写的是 `mode == "keyword"`，但全库其他地方（config、matcher、main）统一用 `"keywords"`。效果是关键词模式下推送消息永远走 AI 格式分支，分数不显示。

同样的问题也藏在 `setup.html` 的 radio value 里：值是 `"keyword"`，但 `main.py` 检查简历时判断的是 `"keywords"`，导致关键词模式下简历缺失检查完全失效——程序不报错，只是默默匹配不到任何东西。

这类 typo 很难靠肉眼发现，因为程序不崩溃，只是静默走错了分支。

---

### Bug 2：非匹配职位混入定时推送

`save_job` 对所有职位都写 `notified=0`，而 `get_pending_jobs` 拉取全量 `notified=0` 的记录。定时模式下，每次到推送时间，所有今天抓到的职位（不管有没有匹配）都会被打包发出去。

修复很直接：给 `save_job` 加一个 `notified` 参数，非匹配职位保存时传 `1`，等于入库即标记"不用推"。数据库已有记录不受影响，只影响新写入的。

---

### Bug 3：免打扰时段结束后积压职位消失

注释写着"安静时段结束后发送"，但实际上根本没有这个逻辑。安静时段内匹配的职位存入 DB，下一轮 `run_cycle` 开始时 `is_seen()` 返回 True 直接跳过，职位永远停留在 `notified=0` 的死区里。

修复：在进入推送逻辑前先检查并 flush 积压的未通知职位，安静时段结束后第一轮会把欠账补上。

---

### Bug 4：`_load_resume` 锁保护了个寂寞

```python
with _RESUME_LOCK:
    if _RESUME_CACHE is not None:
        return _RESUME_CACHE
# 锁已释放，两个线程都到这里了
config = _load_config()
...
_RESUME_CACHE = content.strip()
```

锁只保护了"检查是否已缓存"这一步，检查完锁就释放了。两个线程同时进入、同时读文件、同时写 `_RESUME_CACHE`——如果其中一个读取失败写入了空字符串，会覆盖另一个成功的结果。修复是把整个读取 + 赋值逻辑移入锁内，真正做到原子操作。

这是一个经典的 TOCTOU（检查时机与使用时机不一致）问题。

---

### Bug 5：HTML 解析器嵌套标签失控

`_Extractor` 用一个布尔值 `_skip` 标记"当前在跳过区域"。问题：遇到 `<nav><script>...</script>...</nav>` 时，`</script>` 会把 `_skip` 置回 False，之后 nav 内剩余的 JS 代码就会被当成正文捞出来传给 AI。

换成深度计数器后，只有最外层的 skip 标签关闭时才回到"正常区域"，嵌套层级任意深都不会误判。布尔值适合"二元状态"，计数器适合"可嵌套状态"，这是选错了数据结构。

---

### Bug 6：别名缓存并发写互相覆盖

多关键词同时触发别名学习时，每个线程独立 `load_cache` → 修改 → `save_cache`，中间没有任何同步。Thread A 读出缓存，Thread B 也读出缓存，A 写入，B 覆盖 A 的结果。加了 `_CACHE_LOCK` 后，读-改-写成为原子操作，后写入的线程会在锁内读到前一个线程的写入结果。

---

### Bug 7：保存配置"吃掉"不在 UI 里的字段

`setup.html` 的 `collectConfig()` 只序列化 UI 上展示的字段，`outreach`、`resume_bot`、`web_ui` 三个 section 根本不在表单里。每次点"保存"，这三块配置就从 `config.json` 里消失——程序继续跑但用的是默认值，端口、API 配置都可能悄悄失效。

修复方案选择在后端合并而不是在前端补齐：保存前把旧 config 中这三个 key 合并进来。这样前端代码不需要知道这些字段的存在，后端兜底保证不丢数据。

---

### Bug 8：`storage.mode` 反复被重置

`collectConfig` 构建 storage 对象时缺了 `mode` 字段。每次保存，`mode` 就从 config 消失。下次再保存时 `old_config["storage"]["mode"]` 不存在，`new_config["storage"].get("mode", "local")` 返回 `"local"`，触发"存储方式切换"的虚假警告。修复就是在 `collectConfig` 里补一行读取现有 mode。

---

### 感想

这轮 review 最大的收获是理解了"静默失败"的危害。这八个 bug 里没有一个会让程序崩溃——它们都是悄悄走错分支、悄悄丢数据、悄悄覆盖缓存。用户看到的现象是"感觉有点不对劲"，但很难定位到根因，因为程序表面上一直在跑。

字符串一致性（Bug 1）这种问题说明枚举值不应该散落在各个文件里硬编码。一个常量文件或枚举类型，能让这类 typo 在写代码时就被 IDE 提示出来，而不是运行时静默走错分支。

锁的正确使用（Bug 4）是并发编程里最常见的坑之一：把锁放在检查语句上，而不是放在整个临界区上，形同虚设。锁应该覆盖"检查 + 操作"的完整序列，任何在锁外完成的操作都可能被其他线程打断。

配置覆盖问题（Bug 7、8）是前后端职责边界模糊的结果。前端只序列化它"知道"的字段，后端直接接受并覆盖全量，两者都没有意识到"存在但不在 UI 里的字段"这个边界情况。解法是在边界处明确策略：前端负责它管的字段，后端负责合并它不管的字段。
