# Job Tracker 功能设计说明

## 一、搜索模块

### 位置设置
- **自动获取**：通过 IP 定位自动识别当前所在城市，误差约 1-5 公里，城市级别精度足够。
- **手动填写**：用户手动输入城市名（如 `Hong Kong`、`Shanghai`），手动模式下 `location` 字段不能为空，否则报错提示。
- **搜索半径**：以城市为中心，设定最大距离（公里），传入 `python-jobspy` 的 `distance` 参数。注意：LinkedIn 的距离筛选基于职位发布地点而非精确坐标，为近似参考范围。

### 搜索关键词
- 支持配置多个关键词，抓取时以 `OR` 逻辑合并搜索。
- 建议填写职位英文名，如 `Python Engineer`、`Backend Developer`。
- 每个关键词可配置**手动别名**（用户直接填写的同义词）。
- 开启别名学习后，系统会自动从 AI 获取更多别名并缓存在本地（详见别名学习模块）。

### 时间范围
- `hours_old`：只抓取指定小时数内发布的职位，避免推送旧职位。

### 黑名单过滤
- **屏蔽公司**：公司名包含黑名单项则直接过滤，支持模糊匹配。
- **屏蔽关键词**：职位标题包含指定词则过滤，如 `intern`、`unpaid`。

### 抓取频率
- `interval_minutes`：抓取间隔（分钟），建议 ≥ 30 分钟，低于 30 分钟时系统会推送风险提示。
- **重试机制**：单次抓取失败后自动重试，可配置最大重试次数和重试间隔。连续失败达到上限后暂停指定分钟数，并通过通知渠道告警。
- **恢复通知**：抓取恢复正常后主动推送恢复消息。

## 二、匹配模块

### 关键词打分模式
- 从用户上传的简历文件（`.txt` / `.pdf` / `.docx`）中提取技能关键词。
- 将技能词与职位描述对比，每命中一个词加 5 分，满分 100 分。
- 分数 ≥ `min_score`（用户配置的最低分）时触发推送。
- 适合求职方向固定、简历技能明确的用户。

### AI 判断模式
- 用户用自然语言描述自己的求职要求和背景（`user_prompt`）。
- 系统将职位信息和用户描述一起发给 AI，AI 直接返回"推送/不推送"和一句话理由。
- 不设分数阈值，判断权完全交给 AI，适合跨领域求职或转行场景。
- **语义优先匹配**：AI prompt 明确要求跨语义等价判断（Developer ≈ Engineer ≈ SWE），跨语言等价判断（同一职位的中英文、日英文写法都算匹配），以及上下位概念识别（用户要"工程师"，"高级工程师"也推送）。
- **60% 门槛**：核心需求满足过半即推送，避免过严过滤导致漏掉合适职位。

### AI Provider 支持
| Provider | 默认模型 | 说明 |
|----------|---------|------|
| Claude (Anthropic) | claude-haiku-4-5-20251001 | 速度快，成本低 |
| OpenAI | gpt-4o-mini | 广泛可用 |
| DeepSeek | deepseek-chat | 成本更低，兼容 OpenAI SDK |

用户可在配置中指定 provider、API Key 和具体模型（留空使用默认）。

## 三、存储模块

### 本地 SQLite（主存储，必须）
- 所有抓取到的职位和去重记录都保存在本地数据库。
- 去重逻辑以本地为准：通过 URL 的 MD5 哈希值（取前12位）判断是否已见过该职位。
- 默认路径：`data/jobs.db`（相对程序所在目录，程序移动后自动适配）。
- 断电重启后数据不丢失，稳定可靠。

### Google Sheets 同步（可选备份）
- 启用后，每条新匹配职位会在保存本地的同时同步一份到 Google Sheets。
- 需要配置 Spreadsheet ID 和 Google Cloud Service Account 凭证文件。
- 用途：云端备份、多设备查看历史推送记录。
- **注意**：切换存储方式时历史去重记录不会迁移，系统会推送提示消息告知用户。

## 四、通知模块

### 推送时机
- **实时模式**：每次抓取完成后，立即推送符合条件的新职位。
- **定时模式**：符合条件的职位暂存在数据库，到达用户配置的推送时间点时统一汇总推送。可配置多个时间点（如每天 09:00、13:00、18:00），适合不想被频繁打扰的用户。

### 免打扰时段
- 可选开启，配置开始和结束时间。
- 时区自动根据用户 IP 识别，也可手动指定（支持标准时区名如 `Asia/Shanghai`、`America/New_York`），适合在一个时区生活但找另一个时区工作的用户。
- `hold_and_send`：免打扰期间暂存的职位，在免打扰结束时统一推送（可关闭此选项则直接丢弃）。

### 通知渠道

**Telegram**
- 主要通知渠道，使用 HTML 格式（`parse_mode=HTML`），比 Markdown 更稳定。
- 每条推送包含：职位名、公司、地点、发布时间、匹配分数/AI 理由、职位链接。

**Email（可选）**
- 通过 SMTP 发送 HTML 邮件，支持 Gmail 等标准 SMTP 服务。
- 与 Telegram 可同时开启，互不影响。

### 系统心跳
- 每 24 小时推送一次"运行正常"确认消息，防止系统静默崩溃而用户毫不知情。

## 五、Web UI 配置界面

### 访问方式
- **Mac**：双击 `Double Click to Start_Mac.command`
- **Windows**：双击 `Double Click to Start_Windows.bat`

自动启动配置界面并在浏览器打开 `http://localhost:8080`。首次启动会自动创建 Python 虚拟环境并安装所有依赖。

### 功能 Tab
| Tab | 内容 |
|-----|------|
| 搜索设置 | 位置、关键词（含手动别名编辑）、时间范围、黑名单、抓取频率 |
| 匹配设置 | 模式选择、分数阈值、AI 配置 |
| 存储设置 | 本地路径、Google Sheets 绑定、简历生成器接口 |
| 通知设置 | 推送时机、免打扰、Telegram、Email |
| 别名学习 | AI 别名学习开关、缓存查看、手动触发学习 |
| 简历上传 | 上传 .txt/.pdf/.docx，支持拖拽 |
| 职位列表 | 查看/排序/筛选/操作所有匹配职位，含垃圾箱和导出 |

### 操作流程
1. 在各 Tab 中填写配置
2. 点击"保存配置"写入 `config.json`
3. 点击"启动 Job Tracker"开始运行主程序
4. 后台进程独立运行，关闭配置界面不影响主程序

## 六、职位列表模块

### 查看与排序
- 表格展示所有匹配职位：职位名（含链接）、公司、地点、发布时间、匹配分数/AI 结果。
- 排序维度：发现时间、匹配分数、公司名称、发布时间，支持升序/降序切换，点击表头快速切换。

### 筛选
- 公司名模糊搜索
- 投递状态精确筛选（未处理 / 已投递 / 不感兴趣）
- 发现日期范围筛选
- 仅显示收藏职位

### 每条职位操作
- ⭐ **收藏**：标记特别感兴趣的职位，可单独筛选查看
- **状态切换**：未处理 / 已投递 / 不感兴趣，下拉即改
- **备注**：自由文本备注，失焦自动保存
- 🗑 **移入垃圾箱**：软删除，不影响去重记录

### 垃圾箱
- 独立 Sub-Tab 查看已删除职位
- 支持恢复或永久删除（永久删除需二次确认）

### 导出
- 格式：CSV（可直接用 Excel 打开）或 JSON（供程序对接）
- 导出当前筛选结果，浏览器直接下载到本地
- 可选导出垃圾箱内容

## 七、简历生成器接口

### 设计原则
- 默认关闭，启用后对现有推送流程零影响
- **实时触发**：每次匹配到职位立即输出，不受定时推送模式影响
- 分数不传递（对简历生成器无意义），只传职位核心信息

### 输出内容
```json
{
  "title": "Senior Python Engineer",
  "company": "Airwallex",
  "location": "Hong Kong",
  "link": "https://linkedin.com/jobs/...",
  "description": "完整职位描述...",
  "posted_at": "2026-03-21"
}
```

### 输出方式

**本地文件模式**
- 每个匹配职位写入一个 JSON 文件，文件名含时间戳和职位信息
- 输出目录用户自定义，默认 `~/.job-tracker/matched_jobs/`
- 简历生成器监听此目录，有新文件即处理
- 适合两个程序跑在同一台 Mac 的场景

**Webhook 模式**
- 匹配到职位后立即 POST 到用户填写的 URL
- 适合简历生成器部署在其他机器或云服务器的场景

## 八、关键词别名学习模块

### 设计目标
LinkedIn 搜索基于关键词精确匹配。用户配置了"Software Engineer"，但 LinkedIn 上相同职位可能写作"Developer"、"SWE"、"开发工程师"，全部会被漏掉。别名学习模块的目标就是自动扩展搜索词覆盖范围。

### 缓存架构
别名数据存储在 `data/alias_cache.json`，格式：
```json
{
  "keywords": {
    "Software Engineer": {
      "en": ["Developer", "SWE", "Programmer"],
      "zh": ["软件工程师", "开发工程师"],
      "last_updated": "2026-03-22"
    }
  }
}
```

### 语言感知
学习哪些语言的别名，由搜索位置决定：
- 英文**永远**包含（默认语言）
- 从位置字符串识别附加语言（如 `Hong Kong` → 中文；`Germany` → 德文）
- 抓取结果中出现未预期语言的 JD → 后台补学该语言

### 触发机制

| 触发条件 | 行为 |
|---------|------|
| 用户新增关键词（保存配置） | 立即后台学习新关键词的别名 |
| 每次抓取前 | 检查缓存是否过期（超过 `refresh_days` 天），过期则后台刷新 |
| 发现新语言 JD | 后台针对该语言补充学习 |

所有学习操作在独立守护线程中运行，不阻塞主程序。

### 手动别名与 AI 别名
两者可并用：
- **手动别名**：用户在搜索设置 Tab 中直接添加，权重最高，永不自动删除
- **AI 缓存别名**：自动学习，可定期刷新，用户可在别名学习 Tab 中查看和删除
- **AI 建议**（即时）：在别名学习 Tab 点击「AI 建议」，即时获取推荐（不写入缓存）

### Token 消耗估算
每个关键词每种语言一次学习约 200–400 Token。以 DeepSeek 计，约 $0.0001。缓存后不重复调用，7 天才刷新一次，日常运行额外成本接近零。

## 九、适用场景说明

| 场景 | 推荐配置 |
|------|---------|
| 方向固定，有简历 | 关键词模式 + 上传简历 |
| 跨领域 / 转行 | AI 模式 + 自然语言描述 |
| 不想被频繁打扰 | 定时模式 + 配置2-3个推送时间 |
| 跨时区找工作 | 免打扰 + 手动设置本地时区 |
| 需要历史记录备份 | 开启 Google Sheets 同步 |
| 与朋友共用代码 | 各自维护 config.json，独立运行（所有数据都在各自的 data/ 目录） |
| 需要追踪投递进度 | 职位列表 Tab 标记状态 + 备注 |
| 对接简历生成器 | 开启简历生成器接口，选择输出方式 |
| 搜索覆盖面不足 | 开启别名学习，程序自动补充多语言 / 多行业的同义词 |
| 在非英语地区求职 | 开启别名学习，根据求职城市自动学习对应语言别名 |
